# MyWiki 工程红线与开发准则

本文档定义 `mywiki` 开发过程中必须遵守的架构约束、工作流要求与清理标准。

它是本仓库开发规则的单一事实来源。若与更高优先级的系统/开发者/用户指令冲突，以更高优先级为准；否则以本文为准。

---

## 零、统一治理与审批门禁

### 1. 方案先行准则
- 任何涉及知识库结构、页面类型、CLI 语义、存储模型、导出格式、查询/合并规则的变更，必须先形成方案或设计草案。
- 用户明确回复“批准”、“同意”、“执行”后，方可开始实现。
- 未经审批擅自引入的结构性变更，视为无效交付。

### 2. 共享规则归一化
- `方案先行准则`、`六步走战略`、`wiki 自动生成约束`、`迁移收口与技术债治理规则` 以本文为唯一事实来源。
- `AGENTS.md`、`README.md` 与其他入口文档应优先引用本文，而非复制整段规则。
- 规则更新时，先更新本文，再同步引用说明。

## 一、推荐开发流程：六步走战略

### 1. 契约先行
- 在编写业务逻辑前，先明确要变更的契约：页面 frontmatter、CLI 参数、存储字段、导出格式、报告结构或健康检查输出。
- `system/schemas/`、导出的 markdown 结构、CLI 行为与测试用例共同构成本项目的契约层；禁止“代码已变，契约未变”。
- 若未来引入 API，则 API 契约必须有唯一事实来源，不能与 markdown 文档、脚本、手写类型长期并行。

### 2. 单体骨架
- 严禁循环依赖。
- 当前推荐分层如下：
  - `config/fs/markdown`：基础工具与渲染
  - `ingest/query/lint/export`：领域流程能力
  - `db`：存储适配器
  - `services`：工作流编排与规则执行
  - `cli`：用户入口
- 底层模块不得反向依赖上层入口。

### 3. Fixture/Mock 驱动
- 在外部依赖未就绪时，优先使用本地 fixture、`createInMemoryRepositories()`、`mongodb-memory-server`、原始 markdown 样本驱动开发。
- 禁止让核心工作流绑定单一外部环境后才可验证。

### 4. 垂直切片
- 一次做通一个完整工作流：`raw -> ingest -> state -> wiki export -> meta artifacts -> doctor/repair/test`。
- 禁止只做半层实现后把一致性问题留给后续阶段。

### 5. 可观测性
- 任何改变仓库状态的工作流，都应留下可追踪证据：`meta/log.md`、批次报告、lint 报告、doctor 输出或测试断言。
- 对于冲突、漂移、去重、跳过、修复等行为，必须显式报告，禁止静默吞掉差异。

### 6. 自动化防护网
- 涉及 CLI 行为、导出结构、存储适配、query merge、doctor/repair 的修改，必须有自动化测试覆盖。
- 默认验证命令是 `npm test`。
- 若修改 mongo 适配或跨存储一致性，还必须补或运行 mongo 集成测试。

## 二、技术工程红线

### 1. Wiki 与原始资料红线
- `wiki/` 是自动导出层，不允许作为手工编辑真相源。
- `raw/` 是原始资料层，不得随意重写或删除既有原始材料。
- 若需要修改 wiki 内容，必须通过 CLI / service / ingest / repair 流程完成。

### 2. 单一真相源
- 同一职责只允许一个主实现 / 主入口 / 主注册点。
- `file` 与 `mongo` 可以同时存在，但它们只是不同存储后端，不能演变成两套并行业务规则。
- 页面、查询、比较、时间线、lint、doctor、repair 的规则必须在同一工作流层维护，禁止在多个入口各写一套。

### 3. 存储适配红线
- CLI、查询、导出与 lint 不得直接操作 `meta/manifests/state.json` 或 Mongo collection；必须通过 repository adapter。
- 新增存储字段时，必须同时考虑 `file` 与 `mongo` 两种模式。
- 涉及跨存储一致性的改动，必须能通过 `doctor` 或测试验证。

### 4. 导出与漂移红线
- 导出的页面必须保持稳定结构，frontmatter 与章节顺序不可随意漂移。
- 冲突、来源不足、内容漂移、相似但未合并的问题必须显式标记，而不是被隐藏。
- `repair` 只能重建或清理导出层，不能偷偷重写原始资料层。

### 5. 输出与可读性
- 用户可见报告必须可扫描：批次报告、lint 报告、doctor 输出要有明确计数和命名。
- 禁止把“只统计数量”当成完整健康检查；可定位的问题应尽量给出具体 slug、文件名或集合名。

## 三、迁移收口与技术债治理规则

### 1. 迁移必须自带收口条件
- 引入新目录、新入口、新规则、新适配层前，必须先说明最终保留对象是谁。
- 任何迁移都必须明确：
  - 最终真相源
  - 兼容层范围
  - 调用方位置
  - 验证命令
  - 旧层退场条件

### 2. 兼容层只允许薄封装
- shim、re-export、兼容命令、过渡字段只能承担最薄一层转发，不能沉淀长期业务逻辑。
- 临时命名、实验目录、旧入口别名必须有明确 owner 和退出条件。

### 3. 禁止机械拆分
- 不允许以 `part1/part2` 之类的体积切分替代职责切分。
- 文件拆分必须按语义边界、依赖边界、工作流边界完成。

---

## 四、清理 / 删除决策标准

### 1. “未引用”不等于“可删除”
- 禁止仅凭静态搜索、lint 告警、IDE 提示或单文件未使用就直接删除对象。
- 删除前必须同时确认：
  - 代码路径上无路由、脚本、导出、兼容层、运行时映射依赖
  - 功能树状态明确为重复冗余或正式下线

### 2. 默认保守
- 无法证明可删时，默认不删。
- 优先选择：补注释、补文档、标记 deprecated、登记技术债、在报告中注明保留原因。

### 3. 清理必须可说明
- 凡提交含删除/裁剪动作，汇报中必须说明：
  - 清理对象
  - 所属功能节点
  - 状态判定
  - 删除依据
  - 若未删，保留原因是什么

## 五、指标、审计与报告口径

### 1. 数字必须可追溯
- 报告、doctor、lint、batch ingest、方案与阶段总结中的关键数字必须可回溯到命令、测试、文件或时间点。
- 无法复现的数字，不得写成“当前值”。

### 2. 历史快照不得冒充当前状态
- 旧测试通过数、旧覆盖率、旧 lint/doctor 结果，若未重新测量，必须标注日期或说明是历史快照。

### 3. 口径必须一致
- 总览、正文、报告之间若采用不同统计口径，必须显式说明。
- 不允许同一文档内部出现互相冲突的“当前状态”。

---

## 六、与现有仓库规则的关系

- `AGENTS.md` 负责给 agent 指出仓库目标、目录规则、page types 与 ingest/query/lint 行为。
- 本文负责约束更高层的开发流程、治理规则、清理标准与一致性要求。
- 对本仓库进行任何写操作前，应先读取 `AGENTS.md` 与 `STANDARDS.md`。
